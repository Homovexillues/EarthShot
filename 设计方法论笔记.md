# 软件设计方法论笔记

> 一份从"不懂怎么设计软件"到"开始拥有设计直觉"的对话整理。
>
> 起因是 EarthShot 项目的 [设计文档.md](./设计文档.md) 已经成型，但我意识到自己只提了最初的需求——"做一个轻量的、类似 Flameshot 的、专注框选识别 QR 码的工具"——整个设计是 Claude 推导出来的。我想知道：**这种设计能力是怎么来的？怎么靠自己脑子做到？**

## Part 1：从需求到设计的方法论

### 1.1 起点：把愿景压成一句话

设计的第一个动作不是画架构图，而是把需求压成最短的描述。EarthShot 的版本是：

> 一个轻量、内存内完成、可扩展的 Avalonia 截图工具，框选即识别 QR 码。

这句话里每个词都不是装饰：

- "轻量" → 不做成功能堆叠的工具
- "内存内完成" → 不写盘，隐私+性能+简化代码三重收益
- "可扩展" → 暗示要为未来 OCR/取色等留口子
- "框选即识别" → **唯一**的 MVP 路径

**配套动作：写"非目标"清单**。决定不做什么比做什么更难，但能省掉 80% 的设计漂移。新人最常犯的错就是在 MVP 里偷偷塞标注、设置面板、多语言——每塞一个，工期翻倍。

> **关键心法**：好设计的标志不是"看起来完整"，而是**诚实地暴露未知**——主动把"暂不考虑"和"已知风险"写进设计文档（参见 [设计文档.md §10、§11](./设计文档.md)）。

### 1.2 最难的一步：发现核心抽象

**方法：找"动词三元组"**——把需求场景写成动作链，看哪些步骤是不变的、哪些是变的。

```
[框选屏幕区域] → [识别 QR 码]    → [复制到剪贴板 + 提示]
[框选屏幕区域] → [OCR 文字]      → [复制文字]
[框选屏幕区域] → [取一个像素颜色] → [复制 HEX]
[读剪贴板图片] → [识别 QR 码]    → [打开浏览器]
[选窗口截图]   → [无处理]        → [保存到文件]
```

竖着看：

- 第一列 → "**取像素**"（来源不同，产出都是像素块）
- 第二列 → "**对像素做某种解读**"
- 第三列 → "**用结果做点事**"

这就是 EarthShot 的 `Capturer → Processor → Action` 三段管道的来源（[设计文档 §2](./设计文档.md)）。

> **关键心法**：你不是在"设计架构"，你是在**给业务流程命名**——从"具体场景"中归纳出"哪些步骤本质上是同一类事"。

> **抽象粒度判断口诀**：每一层抽象都要能回答"什么变化在这层背后被吸收？"。回答不上来就是过度抽象。

### 1.3 接口与数据类型：稳定 = 价值

`CapturedImage` 用 `ReadOnlyMemory<byte>` 而不是 `Avalonia.Bitmap` 或 `SkiaSharp.SKBitmap`，是一个刻意选择。

设计原则：

1. **不依赖具体技术**——返回字节流而非 UI 框架类型，让 Core 不依赖任何 UI 库
2. **接口要瘦**——`IScreenCapturer.Capture` 只接收一个 `PixelRect`，不接收"文件名""保存格式"等会随实现变的参数
3. **数据类型要哑**——`record` 只描述数据结构，不带行为，AOT 友好

> **接口质量口诀**：返回类型越**贴近某个具体技术**（`Avalonia.Bitmap`、`System.Drawing.Image`），吸收能力越差；越**通用、越原始**（字节流 + 元数据），吸收能力越好。

> **新手陷阱**：往接口里塞"以防万一"的参数。每多一个参数，所有实现都得处理它，所有调用方都得提供它。**接口的成本是 N×M（N 个实现 × M 个调用方），不是 1**。

### 1.4 项目结构：YAGNI 决策

EarthShot 只分 `Core + App` 两个项目（[设计文档 §5](./设计文档.md)），不是 `Domain / Application / Infrastructure / Presentation` 四层。

判断"该不该拆"的方法：列出拆包能带来的具体收益。

| 候选拆法 | 具体收益 | 现在需要吗？ |
|---|---|---|
| Core + App | Core 可 headless 测试，未来可独立打包 | ✅ 立即就能用 |
| Core.Capturing.Windows | 多平台时换实现 | ❌ MVP 只支 Windows |
| Core.Processing.Qr | QR 模块独立发 NuGet | ❌ 没人会单独用 |
| Core.Abstractions（只放接口） | 第三方实现接口而不依赖实现 | ❌ 没第三方 |

→ 只有 **Core + App** 这一刀有立即收益，其他都是"为想象中的未来"做的工作。

> **YAGNI 黄金问题**：「如果不拆，未来真要拆时成本是多少？」在 .NET 里把一个文件夹提取成新项目大概 10 分钟。所以等"真要拆"的那天再说，比现在多养 N 个空项目划算得多。

### 1.5 实施路线的两条原则

**原则 1：每个里程碑必须"端到端可见"**

- ❌ 反例排序：把所有接口写好 → 写所有 Core 实现 → 写 UI → 最后接起来。**前 80% 时间都在写不能跑的代码**。
- ✅ 实际排序：M1 透明窗口（**能看到**）→ M2 拖框（**能交互**）→ M3 拿到字节（**能拿数据**）→ M4 识别（**有结果**）→ M5 用结果（**完成闭环**）→ M6/M7 外围。

每个里程碑都能跑、能演示、能让你回答"这条路走得通吗？"。

**原则 2：风险大的事往后放，骨架最先**

- M1 骨架不能往后——后续都建在它之上
- M7 多显示器是已知地雷区——放最后避免同时调试两件事
- M6 全局快捷键和核心功能正交——放后面调试更方便

### 1.6 工时估算：识别风险，而非预测时间

| 里程碑 | 工时 | 为什么是这个数 |
|---|---|---|
| M1 | 1 晚 | 跟着 Avalonia 模板做，纯按部就班 |
| M3 | **1-2** 晚 | P/Invoke 有未知 |
| M7 | **1-2** 晚 | DPI 混合缩放是已知坑 |

数字本身不重要——**带不带"区间"才是关键**。固定数字 = 有把握的活；区间数字 = 会踩坑的活。这是给自己的"风险地图"。

> **新手估时法**：先估"如果一切顺利"，然后看每项有几个"未知"。
> - 0 个未知 → 直接乘 1.2
> - 1 个未知 → 乘 1.5–2
> - 2+ 个未知 → 区间表示，上限是顺利的 3 倍
>
> 不是公式，是"承认未知是估时第一变量"的纪律。

### 1.7 可扩展性：用假想需求验证当前设计

> **好的扩展点不是设计出来的，是被未来需求"敲打"出来的**。

具体做法：拿"未来加 OCR""未来上传图床"等假想需求去试现在的接口——能不能只新增一个实现类 + DI 注册就完成？

| 假想需求 | 走当前抽象 | 通过吗 |
|---|---|---|
| 加 OCR | 新增 `IImageProcessor` 实现 + 注册 | ✅ |
| 加取色 | 同上 | ✅ |
| 保存到文件 | 新增 `IResultAction` 实现 | ✅ |
| 上传图床 | 同上 | ✅ |
| 截特定窗口 | 新增 `IScreenCapturer` 实现 | ✅ |
| **加标注（编辑）** | 要改 OverlayWindow 状态机 | ❌ |

最后一项**故意**不通过——这是诚实地承认当前抽象的边界。**别现在就为它增加扩展点**，等需求真来了再讨论是否值得。

> **铁律**：扩展点不是越多越好。每个扩展点都是终生债务（要测试、要维护、要文档）。规则是**第三次出现相同变化时再抽象**（Rule of Three）。

## Part 2：三个核心概念的深入

Part 1 中"决策权归属"是整个方法论的内核。它和另外两个概念——"独立变化的需求"和"变化在这层背后被吸收"——其实是**同一件事的三个角度**。

### 2.1 决策权归属

不要把"模块划分"理解成"代码住在哪个文件夹"，要理解成"**这件事谁说了算**"。

**餐厅类比**：

- 采购决定"从哪买菜"
- 厨师决定"菜怎么做"
- 服务员决定"怎么端给客人"

如果一个人同时干三件事，痛点不是"忙不过来"，而是"**搅在一起**让每个简单的决定都互相干扰"——想换供应商时正在颠勺，客人投诉时正在切菜。

回到 EarthShot：

```
Capturer    决定"像素从哪来"      —— 不关心拿到后干嘛
Processor   决定"像素怎么解读"    —— 不关心像素从哪来、结果给谁
Action      决定"结果怎么用"      —— 不关心结果怎么算的
```

> **核心价值**：当你写到一半发现 `QrCodeProcessor` 里出现了"复制到剪贴板"的代码——立刻能反应过来：「不对，剪贴板怎么用是 Action 的决策权，Processor 越权了」。这种警觉只有从"决策权"角度才会有，从"模块划分"角度不会。

### 2.2 独立变化的需求

> **真正的关键不是"会不会变"，而是"两件事会不会一起变"**。

判断方法：把可能的需求两两配对，问自己「这两条需求**会同时出现在一个 PR 里**吗？」

| 需求 A | 需求 B | 会一起改吗？ |
|---|---|---|
| 加 OCR 识别 | 把结果发到 Slack | ❌ |
| 加 OCR 识别 | 升级 Avalonia 版本 | ❌ |
| 改 QR 库版本 | 改剪贴板写入格式 | ❌ |
| 改 `CapturedImage` 加 `Stride` | 调整像素格式 | ✅ |

> **核心规则**：**会一起改的，放一起；几乎从不一起变的，分开**。

EarthShot 三层划分推导：

- "换捕获方式" / "换识别能力" / "换结果用途" → 在过去 20 年截图工具历史里**从来不是一起发生的** → 必须分层
- `Pixels` / `Width` / `Height` / `Stride` 字段 → 改图像格式时**总是一起改** → 不再分

> **新手错误**：把"看起来不一样的东西"分开，而不是把"会一起变的东西"放一起。

### 2.3 变化在这层背后被吸收

每个抽象层都像**一道墙**。墙的作用是**让墙后的变化不传到墙前**。

```
墙前 (调用方):  ScreenshotPipeline.RunAsync(...)
   ↓ 只看见接口
墙   (接口):    IScreenCapturer.Capture(region) → CapturedImage
   ↓ 接口背后是实现
墙后 (实现):    GdiScreenCapturer        ← MVP 用 GDI BitBlt
              WindowCapturer            ← 未来截特定窗口
              DxgiScreenCapturer        ← 未来用 DXGI 提速
```

实现可以换三种，墙前一行不改——这种"看不见"叫**吸收**。

**自测题**：把 `GdiScreenCapturer` 整个删掉换成 `DxgiScreenCapturer`，多少别处代码要改？

- 0 处 → 抽象设计得好 ✅
- 5 处 → 接口有"泄漏"
- 50 处 → 等于没分层

**真实的"吸收失败"案例**：如果当初接口设计成 `Bitmap Capture(...)` 返回 Avalonia 类型，DXGI 实现就被迫做"GPU 字节流→Bitmap"的包装——这是变化没被完全吸收，跑出来污染了实现。

而当前 `ReadOnlyMemory<byte> + 元数据` 的设计，**任何来源**（GDI、DXGI、内存图、文件）都能直接产出，不用做转换。

### 2.4 三者关系

| 概念 | 视角 | 在问什么 |
|---|---|---|
| **决策权归属** | 组织视角 | 这件事**谁说了算**？ |
| **独立变化** | 时间视角 | 这两件事**会不会同时变**？ |
| **变化被吸收** | 传播视角 | 改这里**会不会影响别处**？ |

三个测试都通过 = 好抽象。

### 2.5 实战审查法：三把尺子

**场景**：有人提议在 `IScreenCapturer` 里加方法：

```csharp
public interface IScreenCapturer
{
    CapturedImage Capture(PixelRect region);
    void SaveToFile(CapturedImage image, string path);  // 新增
}
```

逐条审查：

1. **决策权归属**：保存到文件是"取像素"的决策权范围吗？ → 不是，那是 Action 的事
2. **独立变化**：保存格式（JPG/PNG）会跟"换捕获方式"一起改吗？ → 不会
3. **变化被吸收**：以后想改"保存到云"，会不会被迫修改所有 `IScreenCapturer` 实现？ → 会

→ 三条都失败 → 应新建 `SaveToFileAction : IResultAction`，不动这个接口。

## Part 3：跨学科视角——和钱学森系统工程的关系

整理这套方法论时，我注意到"决策权归属"这种思维和组织管理、和钱学森的《工程控制论》《论系统工程》**思想高度相似**——这不是巧合。

### 3.1 为什么这个联想成立

钱学森的两本书背后有共同源头：**Wiener 的 Cybernetics（1948）+ Bertalanffy 的 General Systems Theory（1950s）**。这两支学派提出：

> 凡是由"互相作用的部分组成、对环境产生功能"的东西，都遵循一组共同规律——不论它是导弹、生物体、企业还是程序。

软件工程领域有自己的独立线索：Parnas 的"信息隐藏"（1972）→ OO 封装 → 领域驱动设计 → 六边形架构。**没引用钱学森，但结论高度同形**——这本身就是这组原则**普适**的证据。

### 3.2 具体概念对应

**工程控制论 → 接口 / 抽象层**

控制论核心概念"传递函数"：你不需要知道子系统内部，只要知道**输入→输出**契约，就能把它接入大系统。

`IScreenCapturer.Capture(region) → CapturedImage` 就是软件意义上的"传递函数"。Part 2 讲的"墙后变化被吸收" = 控制论里的子系统封装。

**系统工程 → 模块层次 / 总体设计**

| 系统工程术语 | 软件工程对应 |
|---|---|
| 系统的层次结构 | 模块/组件/层的层级 |
| 总体设计部 | 架构层 / 设计文档 |
| 总师 + 主任设计师责任制 | 架构决策权 + 模块所有者 |
| 总体协调 vs 分系统专攻 | 跨模块接口约定 vs 模块内部实现 |
| 阶段验证（型号研制三阶段） | 里程碑式开发（M1–M7） |
| 综合集成 | 集成测试 + 端到端验证 |

EarthShot 的 [设计文档.md §5+§8](./设计文档.md) 本质上是一份小型的"总体设计"文件。

**"决策权归属"确实是系统工程核心**

钱学森参与中国航天时建立的"总师责任制"——总师对总体拍板，主任设计师对分系统拍板，互不越权——和软件里"架构师定接口、模块作者定实现"是同一思想。

更深一层——**Conway 定律**（"软件结构会镜像其组织结构"）说的就是：**系统的功能划分 ≈ 决策权的划分 ≈ 组织的划分**。三者在大型工程里**永远一致**。这正是钱学森把"工程"和"组织"放在同一框架下讨论的根本原因。

**反馈环 → 软件的迭代 / 调试**

调试器=实时反馈，测试=周期反馈，CI=自动化反馈，用户数据=长周期反馈。设计的好坏很大程度取决于"出问题→发现→纠正"的反馈延迟。

### 3.3 哪些不能直接套用

**1. 数学工具不直接转移**：工程控制论核心是微分方程描述的连续系统；软件是离散事件系统。**概念可借用，数学不必硬搬**——你不会拿 Laplace 变换分析一个 `IScreenCapturer`。

**2. "全面规划"和现代 YAGNI 有真实冲突**：

- 钱学森时代的中国航天 = 国家计划体制下的大工程，需求/预算/时间确定 → "总体设计先行"是对的
- 现代软件小工具 = 需求会变、未来不确定 → "总体设计先行"会变成过度设计

> **本质差别**：钱学森的世界确定性高、规模大，宁可前期多投入设计；现代软件确定性低、迭代快，宁可少设计、勤迭代。**不同环境的最优解，不是对错关系**。

**3. "开放复杂巨系统"对 MVP 是 overkill**：钱学森晚年针对"还原论失效"的极端复杂系统（人体、社会、生态）提出的方法。EarthShot 这种规模属于"普通复杂系统"，分层+模块化就够用。

**4. 软件需要"自上而下+自下而上交替"**：写代码时会发现"原来设计某些地方不对"需要回去改。软件的"建造成本"远低于"修改成本"，**所以可以容忍多次重设计**——这是软件相对于传统工程的真实差异。

### 3.4 怎么实际用

把这两本书当作**思维词汇库**和**meta 思考训练**，不当软件方法论。

**值得借的核心概念**：

1. **传递函数式思考**——逼自己用"输入/输出契约"看每个组件
2. **层次性**——任何系统都是"系统的系统"，时刻问"我在哪一层、兄弟在哪、父在哪"
3. **总师 vs 分系统**——把"架构决策"和"实现决策"严格分开
4. **反馈与稳定性**——所有循环依赖、重试风暴、配置漂移都是同一类病

**知道存在但不用照搬的**：

5. 总体设计部——大项目可借鉴，小项目过重
6. 三论合一——好的思维地图，但不要硬套
7. 综合集成 / 开放复杂巨系统——研究兴趣可以，别拿来做下个项目的方法论

## Part 4：守则与延伸

### 设计决策五守则

挂在脑子里的清单，下次自己设计时可以逐条问：

1. **删除一个特性比加一个特性更难** → 只加你**能演示当前价值**的东西
2. **三个相似的代码块比一次错误的抽象好** → 重复显眼，错误抽象隐蔽
3. **YAGNI** → 任何"以防万一"的代码，假定它不会被用上，反问"真用上时改造成本是多少"
4. **接口要瘦、数据类型要哑** → 让组合空间最大
5. **端到端 > 分层完美** → 先让一条最窄的路打通到底，再回头加宽

### 自己练这套思维的方法

1. 下次开新项目，**先关编辑器、写目标 + 非目标**。强迫自己列至少 5 条"非目标"。
2. **画动作链**：横着写场景，竖着看哪些列是"同一类事"。
3. **每个抽象问一句**："它背后吸收了什么变化？"答不上 → 删掉。
4. **每个项目拆分问一句**："不拆现在会痛吗？"不痛 → 不拆。
5. **里程碑只接受能 demo 的**。不能 demo 的"中间产物"全部砍掉或合并。
6. **写完设计后，把未来 3 个想得到的需求假装"加进去"**——验证抽象兜不兜得住。

### 一句话收束

> **好软件设计 ≈ 好系统工程 ≈ 好组织设计**——它们都在解决同一个底层问题：**复杂性管理**。
>
> 但每个领域有自己最适合的方法和粒度。**借跨学科的概念框架，但用本领域的具体手法**——是最划算的搭配。

### 延伸阅读

**软件设计的"系统观"经典**：

- 《系统之美》(Donella Meadows, *Thinking in Systems*) —— 系统论的现代普及版，比钱学森更易读、例子更软
- 《设计的结构》(Christopher Alexander, *Notes on the Synthesis of Form*) —— 软件"模式语言"思想的源头

**钱学森**：

- 《工程控制论》—— "传递函数"思维的源头，meta 思考训练
- 《论系统工程》—— "决策权归属""总体设计""阶段验证"的系统化论述

**软件设计正典**：

- Parnas《On the Criteria To Be Used in Decomposing Systems into Modules》(1972) —— "信息隐藏"原始论文，本笔记 §1.3 思想源头
- Eric Evans《领域驱动设计》—— "决策权归属"在大型业务系统里的展开
- Cockburn《六边形架构》—— "墙吸收变化"的具体落地模式
